# The Dynamic Doctrine — what is LLM-decided vs. code

> Authoritative rule for "no hardcoding." Supersedes the "what is hardcoded" list
> in `01-hardcoded-vs-agentic-audit.md` (written before the model registry and the
> Phase 2 agentic-repair path). Date: 2026-06-18 (Phase 3).

## The line

**The LLM owns every ML *decision*. Code owns every *contract, mechanic, and
safety rail*.** A production system cannot be 100% non-deterministic LLM output —
that is the opposite of "never breaks" and "cheap." So "no hardcoding" means **no
hardcoded ML *decisions***, NOT "no code." The table below is the test: if a
constant encodes a *choice that depends on the problem*, the LLM must make it; if
it encodes *how* something is computed or a guardrail bound, it stays in code.

| Concern | Owner | Where |
|---|---|---|
| Task type, target, **metric choice**, good-enough threshold | LLM (decision) | problem_framer |
| Data verdict; which issues matter | LLM (decision) | data_auditor, eda_agent |
| Per-column imputation/encoding/scaling | LLM (decision) | preprocessor |
| Feature hypotheses + formulas | LLM (decision) | feature_engineer |
| Model candidates + initial params | LLM (decision, from the registry) | model_selector |
| Corrective code when a template fails | LLM (decision) | every agent (Phase 2) |
| Notebook structure + narrative | LLM (decision) | exporter |
| metric → sklearn scorer mapping | **code (mechanic)** | `core/metric_registry.py` |
| model name → class / search space | **code (mechanic)** | `core/model_registry.py` |
| RESULT dict + AgentState schema | **code (contract)** | `core/state.py`, templates |
| CV mechanics, pickle format, holdout carve | **code (mechanic)** | data_splitter, agents |
| Sandbox boundary (network/caps/screen) | **code (safety rail)** | sandbox, compose |

## "No predefining under prompts" — how it's enforced

A prompt may NOT hand-type an ML vocabulary or option set. Instead the prompt is
**derived from the single-source registry** so it can't drift and adding an option
is a one-line registry change:

- **Metrics** — the problem-framer's allowed-metric menu is generated by
  `metric_registry.framer_menu_text()` / `all_metrics()`. Adding a metric = one
  registry entry; no prompt edit. (Previously the 13-metric vocabulary was typed
  in the prompt AND duplicated in 5 scoring maps — a documented drift-bug source.)
- **Models** — `model_selector`'s menu is `model_registry.menu_text(task)`.

Heuristic *guidance* in a prompt (e.g. "fraud → recall") is allowed: it shapes the
LLM's reasoning but does not hardcode the outcome — the LLM still chooses from the
goal, and the choice is empirically validated downstream (CV lift, measured CV).

## Guardrails are bounded mechanics, deliberately static

These numeric bounds are NOT per-problem decisions; they are reliability/cost
guardrails and stay in code (tunable via config, not the LLM):

- Iteration gate: `improvement ≥ max(IMPROVEMENT_THRESHOLD, score_std)` — the
  noise-floor half is now **data-driven** (Phase 0.2), not a flat 2%.
- Max 3 iterations; Optuna trial budget (30/20 by size); feature-keep min lift
  (0.002); 5-fold CV; TF-IDF max_features; SHAP top-15; PSI bins; holdout 20%.
- The baseline recipe is intentionally FIXED (LogReg/Ridge) so the floor is
  comparable across runs — a deliberate invariant, not an oversight.
